# ADR-0002: MVP仕様見直し - 食材コピペ登録と買い物リスト計算ロジックの設計

## Status

Accepted

## Context

Issue #38（MVP仕様の見直し）において、コア機能（買い物リスト表示）の実用性を高めるための2つの変更を検討した。

1. **レシピ登録時の食材入力UX**: 手動で食材を1件ずつ入力する方式はUXが悪く、レシピ登録のハードルが高い。コア機能に到達するまでの初期設定コストが大きい。
2. **買い物日の固定**: MVP仕様では買い物日が土曜固定であり、それ以外の曜日に買い物するユーザーにとってコア機能が実質機能しない。

## Decision Drivers

- コア機能（買い物リスト表示）への到達ハードルを下げる
- MVPとして実装コストを最小化する
- 著作権・セキュリティリスクを回避する

## Considered Options

**食材コピペ登録の入力形式**
- テキストペースト → Claude APIでパース
- URL入力 → サーバーでスクレイピング → Claude APIでパース
- 手動入力のみ（現状維持）

**マスタ照合の表記ゆれ対応**
- サーバー側の文字列正規化（カナ統一・全半角統一）
- Claude APIにマスタ照合も一任
- ベクトル類似度検索（Embedding）

**買い物リスト表示タイミング**
- 買い物日当日のみ表示
- 常に直近の買い物サイクルを表示

## Decision

### 食材コピペ登録

**テキストペーストを入力形式として採用する。**

URLスクレイピングは対象サイトの利用規約・著作権上の問題が多く、法的確認コストが大きいためMVP範囲外とする。

登録フロー:
1. ユーザーがレシピサイトの材料テキストをコピーしてペースト
2. サーバー側でグローバルマスタ食材一覧を取得し、Claude APIへ送信
3. Claude APIが食材名の抽出とマスタ照合を一度に処理し、JSON配列で返却（分量は無視）
4. パース失敗時はエラーメッセージを表示し再入力を促す
5. 成功時は確認画面を表示し、ユーザーが確定操作してDB保存
6. マスタ不一致の食材はhousehold-specificとして新規追加（`ingredients.household_id` を利用）

**マスタ照合はClaude APIに一任する。**

文字列正規化はカナ・全半角レベルの表記ゆれしか拾えない。Embeddingは実装コストが高い。Claude APIのパースプロンプトにマスタ一覧を含めることで、サーバー側の実装を最小化しつつ高い照合精度を得る。

即保存は却下。パース結果が常に正確とは限らないため、確認ステップを挟むことでゴミデータの混入を防ぐ。

### 買い物リスト表示

**常に直近の買い物サイクルを表示する。**

買い物日当日のみの表示では、事前確認・計画ができずUXが悪い。

計算ロジック: 今日の日付から `shopping_day` を参照し、直近の買い物日〜次の買い物日の前日を対象期間として、献立→レシピ→在庫を参照して不足食材を表示する。対象期間に献立が未設定の場合はプレースホルダーを表示する。

**`shopping_day` のデフォルトは土曜、NULLなし。**

新規家庭作成時に変更可能なUIを提供する。NULL非許容とすることでエッジケース処理を不要にする。

## Consequences

**Positive:**
- Claude APIにパースと照合を一任することで、サーバー側の実装コストを最小化できる
- 確認ステップによりゴミデータの混入を防げる
- 常時表示により、買い物前日などに事前確認できる
- `shopping_day` のNULL非許容により、エッジケース処理が不要

**Negative:**
- マスタ一覧をプロンプトに含めるため、マスタ増加に伴いトークンコストが増大する
- `ingredients` テーブルへの `household_id` 追加により、既存クエリへのフィルタ追加が必要になる
- URLスクレイピング非対応のため、材料テキストのコピー操作をユーザーに求める
