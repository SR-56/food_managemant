# ADR-0001: ADRによる意思決定記録の運用採用

## Status

Accepted

## Context

AI駆動開発を進めるにあたり、仕様・設計はドキュメント（`docs/`）に、実装内容はコードに残せている。
一方で、過去の意思決定の背景・判断根拠（WHY）が記録されていない状態にある。

AI駆動開発ではセッションをまたぐたびにコンテキストが失われるため、以下の問題が発生しやすい。

- AIが過去に却下した選択肢を再提案する
- AIが当時の制約を知らず、設計の意図と矛盾する実装を提案する
- 人間側も判断の根拠を忘れ、同じ議論を繰り返す

## Decision Drivers

- AIへのコンテキスト補完を仕組みとして確立したい
- 個人開発であり、運用が継続できるシンプルさが必要
- 実装ではなく「なぜその選択をしたか」を残すことが目的

## Considered Options

- ADRを導入する（`docs/adr/` にMarkdownで管理）
- GitのコミットメッセージにWHYを記載する
- 何もしない（現状維持）

## Decision

**ADRを導入し、`docs/adr/` にMarkdownで管理する。**

GitコミットメッセージへのWHY記載は粒度が粗く、意思決定単位での検索・参照がしづらいため却下。
何もしないことはコンテキスト欠落の問題を放置することになるため却下。

## Consequences

**Positive:**
- AIセッション開始時にADRをコンテキストとして渡すことで、過去の判断を再現できる
- 却下した選択肢が明示されるため、同じ議論のループを防げる
- 要件変更時に「この決定は当時の制約に依存していた」と判断できる

**Negative:**
- 意思決定のたびにADRを書くコストが発生する
- ADRが増えるほど管理・参照コストも増える（インデックスの整備が必要になる可能性）
